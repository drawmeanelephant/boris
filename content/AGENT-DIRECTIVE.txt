### CLI truth (Feature 2)

```text
boris                         → HTML site under dist/          (DEFAULT)
boris --html / --html-dir D   → same, explicit
boris --out DIR / --no-rag    → JSON IR only
boris --rag / --rag-dir DIR   → RAG corpus only
boris --jobs N / --watch / --incremental  → HTML helpers (valid alone)
```

There is **no** `zig build rag` step. Correct forms:

```bash
zig build
./zig-out/bin/boris --quiet                    # site → dist/
./zig-out/bin/boris --out .boris --quiet       # IR
./zig-out/bin/boris --rag --quiet              # RAG → rag/
./zig-out/bin/boris --rag-dir ./uploads/rag --quiet
```

Trusted-author HTML: the host adapter allows raw HTML passthrough. Prefer

## Authoring rules (hard — compile will fail if broken)

### Frontmatter — closed set only

Allowed keys **exactly**: `id`, `title`, `parent`, `status`, `tags`.

```md
---
title: My page title
parent: guides/intro
status: published
tags: [guides, graph]
---
```

**Forbidden** (unknown key → `EFRONTMATTER`):

- `parentEntry`, `parent_entry` (export-only name in RAG catalogs — never author FM)
- nested YAML, multiline scalars, anchors, arbitrary keys, unquoted `kind=tip`

Rules of thumb:

- **Trunk** = omit `parent` (or never set it).
- **Satellite** = set `parent: <trunk-entity-id>` where the id is the parent’s
  path-derived id (e.g. `guides/intro` for `guides/intro.md`), unless parent
  used an explicit `id:` override — then match that.
- No satellite-of-satellite, no self-parent, no missing parent, no cycles.
- Entity ids come from path minus `.md` by default (`guides/rag-export.md` →
  `guides/rag-export`). Prefer path-derived ids; only use `id:` when intentional.
- `status`: only `draft` | `published` | `archived` if present.
- `tags`: only `[a, b, "c"]` list form.

### Aside — only registered component

```md
<Aside kind="tip" id="stable-anchor">

Body markdown here. Close tag must start a line.

</Aside>
```

- Kinds: `note`, `tip`, `info`, `warning`, `danger` (default `note`).
- Optional `id`: `[A-Za-z0-9][A-Za-z0-9_-]*` max 64.
- Values **double-quoted**. No nested asides. No `<Figure>`, `<Broside>`, etc.
- Asides stay **in document order** on the page — not graph nodes, not separate HTML pages.
- Inside fenced code, `<Aside` is literal (good for docs-about-docs).

### Markdown / Apex showcase

Use real Unified features where they teach the product:

| Feature | Use in sample content? |
|---------|------------------------|
| Headings, lists, links, emphasis | Yes (baseline) |
| Fenced code + language tags | Yes |
| Tables | Yes (at least one real table) |
| Footnotes | Yes (at least one page) |
| Task lists / nested lists | Yes where natural |
| Math (if sensible for a short aside) | Optional; one small example OK |
| Apex callouts / IAL / fenced divs | Optional; prefer `<Aside>` for product callouts |
| Arbitrary MDX / JSX / executable components | **No** |
| `:::kind` as **authoring** | **No** — that form is RAG **export** only |

### Links and paths

- Site output: `guides/intro.md` → `dist/guides/intro.html`.
- Prefer **site-relative HTML-ish links** that survive publish, e.g.
  `[Introduction](guides/intro.html)` or root-relative style consistent across
  the sample. Do **not** leave author-facing CTAs that only work as raw
  `.md` GitHub links unless you also note “source path.”
- Do not invent a second content root. Default is `content/`.

### Voice and branding

- Product name: **Boris**. Metaphor Load → Roll → Ignite → Reset is fine in
  narrative; do not invent branded component names.
- Honest about phase: HTML default + real Apex shipped; graph-aware HTML nav /
  TOC still roadmap (Feature 6) — don’t claim auto sidebars if layout doesn’t
  have them.
- `layouts/main.html` is minimal chrome (title “Boris Site”, basic admonition CSS).
  Content must read well inside that layout without requiring a design system.

## Target information architecture

Build a small but **coherent docs site**, not a single dump page. Suggested
shape (you may refine names as long as graph rules hold):

```text
content/
  AGENT-DIRECTIVE.txt          # this file — keep; do not compile
  index.md                     # Trunk — home / product pitch (HTML-default era)
  getting-started.md           # Trunk — install, first build, modes table
  guides/
    overview.md                # Trunk — content model + pipeline teaching beat
    trunk-satellite.md         # Trunk or satellite under overview — graph rules
    asides.md                  # Satellite — Aside authoring contract in prose
    apex-markdown.md           # Trunk or satellite — Unified showcase page
    cli-and-modes.md           # Satellite — Feature 2 CLI modes (HTML / IR / RAG)
    rag-export.md              # Satellite — correct RAG commands + corpus map
  reference/
    frontmatter.md             # Trunk — closed keys cheat-sheet for authors
    diagnostics.md             # Optional satellite — exit codes / common errors
```

Minimum viable if you need to ship smaller: **≥6 pages**, **≥2 trunks**,
**≥3 satellites**, covering home, getting started, graph model, Aside, CLI
modes, and Apex showcase.

### Page goals (must land)

1. **Home (`index`)** — What Boris is *now*: Zig docs compiler; default `boris`
   → `dist/`; Apex Unified; Trunk/Satellite; no Node SSG. Clear links into guides.
2. **Getting started** — Zig 0.16 + CMake note (compile-time for Apex),
   `zig build`, first HTML build, `--out` for IR, `--rag` for corpus, `--watch`.
3. **Content model** — Trunk vs Satellite with real examples pointing at sibling
   pages; `parent` only; what fails validation (cycle, missing parent, etc.) in
   prose — do not add broken fixture pages under `content/` that fail the default build.
4. **Asides** — Authoring syntax, kinds, document-order rule, export-only `:::kind`.
5. **CLI modes** — Table for HTML default / IR / RAG; migration from bare-IR habit;
   conflict examples (`--out` + `--rag` → exit 2).
6. **Apex showcase** — One page that is deliberately “pretty markdown”: table,
   footnote, nested list, code fence, at least two `<Aside>` kinds.
7. **RAG** — Accurate commands (`boris --rag`), tree layout, shared graph
   validation, **not** `zig build rag` unless you verify a step exists (it does not).

## Implementation taste

- Prefer fewer excellent pages over many empty stubs.
- Prefer teaching by example (show a table; show an Aside) over only describing them.
- Match existing markdown tone: direct, technical, complete sentences.
- Use `status: published` on finished sample pages; `draft` only if you leave a WIP page and say so.
- After substantive edits, re-run the three modes (HTML / IR / RAG) before declaring done.
- Optional: short bullet under `CHANGELOG.md` `[Unreleased]` if the sample site
  is user-visible enough to mention (“expand demo content under content/”).
